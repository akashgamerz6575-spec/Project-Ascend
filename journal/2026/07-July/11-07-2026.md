# Exercises Solved

### Exercise 1: Predict Function Execution Order
We analyzed a program with three functions to predict control jump flow.
```cpp
#include <iostream>
void first() {
    std::cout << "A ";
}
int main() {
    std::cout << "B ";
    first();
    std::cout << "C\n";
    return 0;
}
```
- **Output:** `B A C`
- **Key Lesson:** The CPU executes `main()`, encounters the function call `first()`, pushes `first` onto the stack, executes `first()`, returns control to `main()`, and continues main's execution.

### Exercise 2: Local Variables and Frame Isolation
We attempted to access a function-local variable outside its scope.
```cpp
#include <iostream>
void myFunction() {
    int x = 100;
}
int main() {
    myFunction();
    // std::cout << x; // Attempt to access x
    return 0;
}
```
- **Output:** Compilation Error: `x was not declared in this scope.`
- **Key Lesson:** Function-local variables are allocated in the function's own stack frame. They are physically inaccessible and destroyed once the stack frame is popped.

### Exercise 3: Duplicate Identifiers Across Stack Frames
We traced two variables sharing the same name in different scopes.
```cpp
#include <iostream>
void fun() {
    int x = 100;
    std::cout << x << " ";
}
int main() {
    int x = 50;
    fun();
    std::cout << x << "\n";
    return 0;
}
```
- **Output:** `100 50`
- **Key Lesson:** Different functions own separate, isolated stack frames. The compiler assigns distinct memory locations to each `x`, avoiding namespace collisions.

### Exercise 4: Pass by Value Trace
We verified the memory behavior of passing a primitive variable by value.
```cpp
#include <iostream>
void change(int x) {
    x = 100;
}
int main() {
    int a = 5;
    change(a);
    std::cout << a << "\n";
    return 0;
}
```
- **Output:** `5`
- **Key Lesson:** Passing by value copies the value of `a` into a new local parameter variable `x` inside `change`'s stack frame. Modifying the copy does not alter `a`.

### Exercise 5: Pass by Reference Trace
We verified the memory behavior of passing a variable by reference.
```cpp
#include <iostream>
void change(int &x) {
    x = 100;
}
int main() {
    int a = 5;
    change(a);
    std::cout << a << "\n";
    return 0;
}
```
- **Output:** `100`
- **Key Lesson:** Passing by reference using `&` binds the parameter `x` as an alias to the caller's variable `a`. They share the exact same memory box, so modifying `x` modifies `a` directly.

### Exercise 6: Nested Function Calls Stack Trace
We traced a chain of function calls to visualize call stack push and pop operations.
```cpp
#include <iostream>
void third() { std::cout << "3 "; }
void second() { std::cout << "2 "; third(); std::cout << "4 "; }
int main() { std::cout << "1 "; second(); std::cout << "5\n"; return 0; }
```
- **Output:** `1 2 3 4 5`
- **Call Stack Lifecycle:**
  1. `main` is pushed.
  2. `second` is pushed on top of `main`.
  3. `third` is pushed on top of `second`.
  4. `third` finishes and is popped.
  5. `second` resumes, finishes, and is popped.
  6. `main` resumes, finishes, and is popped.
- **Key Lesson:** Stack frames are pushed and popped in LIFO (Last In First Out) order.

### Exercise 7: Active Variables Across Stack Frames
Given the nested call chain from Exercise 6, which variables exist in memory while `third()` is executing?
- **Answer:** If `main()` has `int x`, `second()` has `int y`, and `third()` has `int z`, then `x`, `y`, and `z` all exist simultaneously in memory.
- **Reasoning:** Every active function frame in the call chain remains allocated in stack memory until that function returns. Since `main()` and `second()` are suspended waiting for `third()` to return, their stack frames and local variables are still alive.

### Exercise 8: Infinite Recursion and Stack Overflow
We analyzed the memory consequences of calling a function recursively without a termination condition.
```cpp
void fun() {
    fun();
}
```
- **Reasoning:** Each call to `fun()` pushes a new stack frame onto the call stack. Since there is no base case, no frames are popped. Eventually, the stack memory (1MB or 8MB) is fully exhausted. The program terminates abruptly with a Stack Overflow page fault.

# Misconceptions Corrected

1. **Misconception:** Scope is just a set of arbitrary grammatical rules to memorize.
   - **Correction:** Scope is the high-level representation of runtime stack frame lifetime. Local variables only exist while their owning function's stack frame is active on the stack.
2. **Misconception:** Pass by Value modifies the caller's variable.
   - **Correction:** Pass by value creates a separate copy of the variable inside the function's stack frame. Modifications only affect the copy.
3. **Misconception:** Pass by Reference creates a new variable or copy.
   - **Correction:** Pass by reference creates a compiler-level alias (nickname) sharing the original variable's physical address. No new object is allocated.
4. **Misconception:** Variables with the same name cannot coexist.
   - **Correction:** Identically named variables can coexist simultaneously in different stack frames because they reside at unique offset addresses in isolated frames.
5. **Misconception:** Local variables disappear randomly.
   - **Correction:** Local variables disappear because their containing stack frame is popped off the stack when the function returns, making that memory free.
6. **Misconception:** Recursion is a magic loop.
   - **Correction:** Recursion is not a constant-space loop; it repeatedly invokes new function calls, allocating new stack frames for each call, which can lead to Stack Overflow.

# What Clicked Today

- **Functions create their own workspaces:** Every active function call owns a distinct stack frame.
- **LIFO execution order:** The call stack executes functions like a stack of plates. The last function called is the first to finish.
- **Difference between copy and alias:** Pass-by-value replicates data (photocopy), while pass-by-reference shares the same memory box (alias/nickname).
- **Stack Overflow mechanism:** Infinite recursion fills the call stack space until stack memory is exhausted.
